# User Story 01: Create Sound Sphere

## Story Summary

As a musician or visual artist, I want a browser-based instrument that reacts to MIDI or computer-keyboard performance with synchronized sound and black-and-white geometry animations, so I can improvise and compose audio-visual sketches in real time without installing native software.

## Vision

Sound Sphere is an offline-first web app that turns performance input (MIDI and computer keyboard) into two outputs at once:

1. Procedural audio patterns generated by Strudel.
2. Time-based geometric animations rendered with p5.js.

The core value is immediate performance feedback: every note, velocity change, and controller movement should feel responsive, stable, and intentional.

## Product Goals

1. Provide low-latency MIDI-to-audio and MIDI-to-visual triggering in modern browsers.
2. Provide equivalent low-latency computer-keyboard triggering when MIDI is unavailable.
3. Support a focused 3-octave playable range for predictable mapping behavior.
4. Keep visuals expressive but constrained (black/white, geometry-first) for coherent style.
5. Persist user state locally so sessions can be resumed quickly.
6. Ensure operation without network connectivity once dependencies are installed.

## Non-Goals (Current Scope)

1. No collaborative multi-user sessions.
2. No cloud sync for presets/sessions.
3. No DAW plugin integration.
4. No full MIDI remapping UI (advanced MIDI learn is future scope).

## Primary Users and Scenarios

1. Live performer: connects a MIDI keyboard and uses keys + modulation controls for spontaneous performance.
2. Sound designer: explores note mappings and pattern groupings to discover rhythmic/melodic textures.
3. Visual artist: cycles animation types and studies motion behavior tied to playing dynamics.

## Functional Requirements

1. Detect and connect available MIDI input devices via Web MIDI API.
2. Map MIDI notes 36-71 to deterministic sound patterns and visual triggers.
3. Handle MIDI events:
	- note on
	- note off
	- velocity
	- mod wheel (CC1)
	- pitch bend
	- sustain pedal (CC64)
4. Render one of four animation families with lifecycle management.
5. Limit active animations to preserve frame rate.
6. Persist selected device, animation mode, and relevant session state to localStorage.
7. Provide keyboard shortcuts for mode switching and screen clearing.
8. Support computer keyboard note triggers when no MIDI device is connected.
9. Ensure computer keyboard note triggers produce equivalent output behavior to MIDI note-on/note-off for mapped notes.
10. Route computer keyboard events through the same normalization and dispatch path as MIDI note events.

## Non-Functional Requirements

1. Performance:
	- Visual loop should target 60 FPS under typical load.
	- Trigger-to-feedback latency should feel immediate in normal desktop conditions.
2. Reliability:
	- Missing/disconnected MIDI devices should fail gracefully.
	- Invalid MIDI events should not crash rendering or audio engine.
3. Usability:
	- Core controls (device select, animation select, save/load) must be obvious and reachable.
4. Portability:
	- App should run through Vite in modern Chromium-based browsers with MIDI support.

## System Design Overview

### Architecture

The application is organized into five domains:

1. MIDI Input Domain
	- `src/midi/MidiInput.js`
	- `src/midi/KeyboardInput.js`
	- `src/midi/MidiConfig.js`
2. Sound Domain
	- `src/sound/StrudelEngine.js`
	- `src/sound/PatternLibrary.js`
3. Visual Domain
	- `src/visual/P5Sketch.js`
	- `src/visual/AnimationController.js`
4. Session Domain
	- `src/session/SessionManager.js`
5. Application Orchestration
	- `src/main.js`

### Event Flow (Target Behavior)

1. MIDI event enters through `MidiInput`.
2. Computer keyboard event enters through `KeyboardInput`.
3. Input event is normalized and validated against `MidiConfig`.
4. Normalized event is dispatched in parallel:
	- Sound trigger/update to `StrudelEngine`.
	- Visual trigger/update to `AnimationController` and p5 sketch runtime.
5. Session-relevant state changes are captured by `SessionManager`.

### Mapping Strategy

1. Notes are grouped into ranges to map to pattern categories.
2. Velocity maps to:
	- Audio gain/intensity
	- Visual size/opacity/energy
3. Mod wheel maps to timbral and/or modulation depth behavior.
4. Pitch bend affects currently active notes.
5. Sustain pedal extends note lifecycle where applicable.

## Implementation Plan

## Phase 1 - Foundation and Core Loop (Completed)

Objective: establish working app skeleton with end-to-end MIDI -> sound + visual loop.

Deliverables:

1. Vite setup and app shell.
2. MIDI device detection and event stream.
3. Initial Strudel integration and basic pattern triggering.
4. p5 sketch bootstrap with base animation classes.
5. Core orchestration in `main.js`.

Definition of done:

1. Pressing mapped notes triggers audible output and visible animation.
2. No critical runtime errors in normal play.

## Phase 2 - Sound Pattern System (Current Priority)

Objective: enrich musical variety and consistency of note-to-pattern behavior.

Tasks:

1. Build reusable catalog in `PatternLibrary.js`:
	- Drum-oriented patterns
	- Bass-oriented patterns
	- Melody/arpeggio patterns
	- Texture/effect patterns
2. Formalize pattern assignment rules by note ranges in `MidiConfig.js`.
3. Improve `StrudelEngine.js` for:
	- Stable initialization lifecycle
	- Per-note trigger handling
	- Controller updates (mod wheel, pitch bend, sustain)
4. Add fallback behavior when a pattern is unavailable.

Definition of done:

1. Every mapped note resolves to a valid pattern.
2. Controller changes are audible and stable during sustained play.
3. No silent-fail branch for expected MIDI note inputs.

## Phase 3 - Visual System Expansion

Objective: improve visual expressiveness while preserving style constraints and performance.

Tasks:

1. Finalize and tune four animation types in `P5Sketch.js`.
2. Ensure lifecycle controls in `AnimationController.js`:
	- Spawn
	- Update
	- Fade
	- Cleanup
3. Enforce active animation cap and deterministic eviction policy.
4. Ensure all animation constructors initialize required state to avoid runtime null/undefined errors.

Definition of done:

1. Visual modes can be switched live without app restart.
2. Animation cleanup prevents unbounded memory growth.
3. Rendering remains smooth at typical interaction intensity.

## Phase 4 - Audio-Visual Coupling

Objective: make sound and visual responses feel musically and visually unified.

Tasks:

1. Define shared parameter bridge for key event attributes:
	- velocity -> visual amplitude + audio intensity
	- note value -> spatial position / shape variant
	- controller data -> temporal modulation
2. Add coupling utilities to normalize parameter ranges.
3. Tune responsiveness and decay curves by live-play testing.

Definition of done:

1. Fast/soft playing produces clearly distinct visual and sonic behavior.
2. Parameter response is smooth and free from abrupt jumps.

## Phase 5 - Session Experience and UX Polish

Objective: improve user flow for repeated sessions and reduce setup friction.

Tasks:

1. Expand `SessionManager.js` payload schema (versioned).
2. Add UI behavior in `main.js` for:
	- save current state
	- load previous state
	- reset to defaults
3. Provide clear status/error messages for device and session actions.

Definition of done:

1. User can save, reload, and continue a session without manual reconfiguration.
2. Storage schema changes are backward-compatible or safely migrated.

## Phase 6 - Hardening, QA, and Documentation

Objective: ensure production readiness for reliability, performance, and maintainability.

Tasks:

1. Add regression checks for MIDI routing and animation lifecycle.
2. Validate offline behavior after initial dependency install.
3. Profile performance under sustained event load.
4. Update README and usage docs with troubleshooting guidance.

Definition of done:

1. No critical defects in core interaction path.
2. Build pipeline (`npm run build`) completes cleanly.
3. Docs reflect implemented behavior and known limitations.

## Testing and Validation Plan

1. Manual integration checks:
	- Connect/disconnect MIDI device during runtime.
	- Play across full 3-octave range.
	- Trigger rapid note bursts and sustained chords.
2. Keyboard parity checks:
	- Trigger mapped notes using normal computer keyboard input.
	- Verify sound pattern, animation trigger, and lifecycle behavior match MIDI-triggered output for the same mapped note.
	- Verify fallback usability when no MIDI device is connected.
3. Controller behavior checks:
	- Mod wheel sweeps
	- Pitch bend extremes
	- Sustain pedal on/off transitions
4. Visual integrity checks:
	- Mode switch under load
	- Animation cleanup correctness
5. Session checks:
	- Save -> reload -> verify persisted state
	- Behavior after browser refresh
6. Build checks:
	- `npm run dev`
	- `npm run build`

## Risks and Mitigation

1. Browser MIDI compatibility differences.
	- Mitigation: explicit support statement and graceful fallback messaging.
2. Audio init or autoplay restrictions.
	- Mitigation: ensure user gesture triggers engine start.
3. Visual performance degradation under high event rate.
	- Mitigation: animation cap, lightweight geometry, aggressive cleanup.
4. Runtime errors from partially initialized animation objects.
	- Mitigation: constructor-level defaults and defensive guards.

## Acceptance Criteria

1. A user can connect a MIDI keyboard and trigger synchronized audio + visuals immediately.
2. A user can trigger mapped notes with a normal computer keyboard when MIDI is unavailable.
3. For mapped notes, computer keyboard triggers produce equivalent sound and visual behavior to MIDI note triggers.
4. All mapped notes in the 3-octave range produce deterministic output.
5. Visuals remain responsive with bounded active animation count.
6. Session state can be saved and restored from localStorage.
7. The app runs locally and functions without internet after setup.
